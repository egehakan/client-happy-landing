1. OVERVIEW
================================================================================

ClientHappy is a web application designed to help agencies, freelancers, and
development teams gather structured feedback from clients on their projects.
It enables clients to vote on screenshots and answer questionnaires about their
preferences and project requirements.

Key Highlights:
- Built with Next.js (App Router), React, TypeScript, and Tailwind CSS
- Uses Turso (SQLite) for database storage
- Uses Vercel Blob for file storage
- Runs locally on Mac machines
- No client authentication required - clients access via shareable links

================================================================================
2. PURPOSE AND USE CASE
================================================================================

PROBLEM IT SOLVES:
When working with clients on web and app projects, gathering feedback can be
challenging. Clients often struggle to articulate their preferences, and
feedback collection is typically unstructured (emails, calls, scattered notes).

SOLUTION:
This application provides a structured, visual way to collect client feedback:

1. VISUAL VOTING: Clients see screenshots and vote Yes/Maybe/No with
   optional comments, making it easy to understand their preferences at a glance.

2. STRUCTURED QUESTIONNAIRES: Custom questions can be created to gather specific
   information about client requirements, preferences, colors, styles, etc.

3. ORGANIZED DATA: All feedback is stored in a database, organized by project,
   page, and section, making it easy to analyze and reference.

TYPICAL WORKFLOW:
1. Create a project for a client
2. Upload screenshots organized by pages and sections
3. Create questionnaire questions (optional)
4. Share a link with the client
5. Client enters their email and votes on screenshots / answers questions
6. Review the feedback in the admin panel

3. CORE CONCEPTS
================================================================================

PROJECT:
A project represents a client engagement. It can be either "web" or "mobile"
type. Each project has a unique URL slug for client access.

Example: "Acme Corp Website Redesign" -> /projects/acme-corp-website-redesign

PAGES:
Pages are the first level of organization within a project. They typically
represent different pages or screens in the app or website.

Examples: "Homepage", "About Us", "Contact Page", "Dashboard"

SECTIONS:
Sections subdivide pages into logical parts. They help organize screenshots
when a single page has multiple areas to review.

Examples: "Hero Section", "Navigation", "Footer", "Sidebar"

SCREENSHOTS:
Screenshots are the images that clients will vote on. They can be:
- Uploaded locally (stored in Vercel Blob)
- Linked from external URLs

Screenshots can be attached to:
- A specific section (most common)
- A page directly (for page-level screenshots without a section)

QUESTIONS:
Questions are form fields that gather additional information from clients. They
can be scoped to:
- The entire website/project (general questions)
- A specific page (page-related questions)
- A specific section (section-specific questions)

VOTES:
Votes are client responses to screenshots. Each vote includes:
- The vote choice: Yes, Maybe (Mid), or No
- An optional comment explaining the choice
- The client's email identifier

RESPONSES:
Responses are client answers to questionnaire questions. They support various
field types including text, files, colors, dates, and more.

6. CLIENT FEATURES
================================================================================

CLIENT ACCESS:
Clients access the application via a shareable link:
/projects/[project-slug]

No account or login is required for clients. They only need to provide an
email address to track their responses.

CLIENT FLOW:

Step 1: EMAIL ENTRY
┌─────────────────────────────────────────────────────────────────────────────┐
│ The client enters their email address. This email is used to:               │
│ - Identify their votes and questionnaire responses                          │
│ - Allow them to update their responses if they return later                 │
│ - Track completion status                                                   │
│                                                                             │
│ The email is saved in browser localStorage for convenience.                 │
└─────────────────────────────────────────────────────────────────────────────┘

Step 2: SELECTION SCREEN (if both voting and questionnaire exist)
┌─────────────────────────────────────────────────────────────────────────────┐
│ If the project has both screenshots and questions, clients choose:          │
│ - "Vote on Designs" - go to the voting interface                            │
│ - "Complete Questionnaire" - go to the questionnaire                        │
│                                                                             │
│ The selection screen shows:                                                 │
│ - Completion badges if they've already completed an activity                │
│ - "New Items" badges if new content was added since they last visited       │
│ - Disabled states if a feature is not available                             │
└─────────────────────────────────────────────────────────────────────────────┘

Step 3: VOTING OR QUESTIONNAIRE
See detailed sections below for each workflow.

Step 4: THANK YOU PAGE
┌─────────────────────────────────────────────────────────────────────────────┐
│ After submitting votes or questionnaire responses, clients see a           │
│ confirmation page thanking them for their feedback.                         │
└─────────────────────────────────────────────────────────────────────────────┘

RETURNING CLIENTS:
When a client returns to the same project:
- Their email is remembered (from localStorage)
- Previous votes are pre-loaded (they can update them)
- Previous questionnaire responses are pre-filled
- They can change any of their responses and resubmit

7. VOTING SYSTEM
================================================================================

PURPOSE:
The voting system allows clients to provide quick, visual feedback on
screenshots. It's designed to be simple and intuitive.

VOTE OPTIONS:
┌─────────────┬─────────────────────────────────────────────────────────────┐
│ YES (👍)    │ The client likes this / approves this direction             │
├─────────────┼─────────────────────────────────────────────────────────────┤
│ MAYBE (➖)  │ The client is neutral / has mixed feelings                  │
├─────────────┼─────────────────────────────────────────────────────────────┤
│ NO (👎)     │ The client dislikes this / does not approve                 │
└─────────────┴─────────────────────────────────────────────────────────────┘

VOTING INTERFACE:
┌─────────────────────────────────────────────────────────────────────────────┐
│ HEADER                                                                      │
│ - Project name and description                                              │
│ - Back button (if coming from selection screen)                             │
│ - Client email display                                                      │
│ - Theme toggle (light/dark mode)                                            │
├─────────────────────────────────────────────────────────────────────────────┤
│ PROGRESS BAR                                                                │
│ - Shows X of Y voted                                                        │
│ - Percentage complete                                                       │
├─────────────────────────────────────────────────────────────────────────────┤
│ SCREENSHOT DISPLAY                                                          │
│ - Large image view of current screenshot                                    │
│ - Title and description                                                     │
│ - Page and section context (e.g., "Homepage / Hero Section")                │
│ - Screenshot counter (e.g., "3 / 15")                                       │
├─────────────────────────────────────────────────────────────────────────────┤
│ CONTEXT OVERLAY (when changing pages/sections)                              │
│ - When navigating to a new page or section, an overlay appears              │
│ - Shows "Page: [Name]" and optionally "Section: [Name]"                     │
│ - Image is blurred during the 2-second overlay display                      │
│ - Helps clients understand the context switch                               │
├─────────────────────────────────────────────────────────────────────────────┤
│ VOTING BUTTONS                                                              │
│ - Three buttons: Yes (green), Maybe (yellow), No (red)                      │
│ - Selected vote is highlighted                                              │
│ - Click to select/change vote                                               │
├─────────────────────────────────────────────────────────────────────────────┤
│ COMMENT BOX                                                                 │
│ - Optional textarea for additional feedback                                 │
│ - "Add a comment (optional)..."                                             │
├─────────────────────────────────────────────────────────────────────────────┤
│ NAVIGATION                                                                  │
│ - Previous button (disabled on first screenshot)                            │
│ - Next button / Submit button (on last screenshot)                          │
├─────────────────────────────────────────────────────────────────────────────┤
│ THUMBNAIL STRIP                                                             │
│ - Scrollable row of all screenshot thumbnails                               │
│ - Click to jump to any screenshot                                           │
│ - Color-coded borders show vote status:                                     │
│   - Green ring: Voted Yes                                                   │
│   - Yellow ring: Voted Maybe                                                │
│   - Red ring: Voted No                                                      │
│   - No ring: Not voted yet                                                  │
└─────────────────────────────────────────────────────────────────────────────┘

SUBMISSION:
- Client must vote on at least one screenshot to submit
- All votes are submitted together in bulk
- If the client has voted before, their previous votes are replaced
- Redirects to thank you page after successful submission

10. AUTHENTICATION SYSTEM
================================================================================

ADMIN AUTHENTICATION:
The application uses NextAuth.js for admin authentication.

USER TYPES:
┌─────────────────────────────────────────────────────────────────────────────┐
│ ENVIRONMENT USERS                                                           │
│ - Defined in AUTH_USERS environment variable                                │
│ - Format: email:password,email:password                                     │
│ - Automatically verified, no email confirmation needed                      │
│ - Good for: quick setup, development, single-admin use                      │
├─────────────────────────────────────────────────────────────────────────────┤
│ SELF-REGISTERED USERS                                                       │
│ - Register via /register page                                               │
│ - Receive email verification link                                           │
│ - Must verify email within 24 hours                                         │
│ - Good for: multi-user setup, team access                                   │
└─────────────────────────────────────────────────────────────────────────────┘

PASSWORD REQUIREMENTS:
- Minimum 8 characters
- At least one uppercase letter
- At least one lowercase letter
- At least one number

LOGIN FLOW:
1. Navigate to /login
2. Enter email and password
3. If environment user: immediate access
4. If self-registered user: must have verified email
5. Session created using JWT strategy

CLIENT ACCESS:
Clients do NOT need authentication. They access via public project links and
only need to provide an email address for identification purposes.

- Projects: name, type, description, slug

- Screenshots: source type, file path/URL, parent reference
- Votes: vote choice, comment, voter identifier
- Questions: field type, scope, options, required status
- Responses: value, file path

SECURITY MEASURES:
- All admin endpoints verify user authentication
